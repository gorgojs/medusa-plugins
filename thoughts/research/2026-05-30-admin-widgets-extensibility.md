# Виджеты в кастомных зонах админки — варианты реализации

Дополняет [2026-05-28-medusa-integration-package-architecture.md](2026-05-28-medusa-integration-package-architecture.md). Там у descriptor'а уже есть `customSections` — но это позволяет **только самому плагину** добавлять секции в **свою** settings-страницу. Здесь решается более широкая задача: дать **сторонним потребителям наших интеграций** внедрять собственные React-виджеты в **наши** кастомные расширения админки, в зонах, которые определяем **мы**.

---

## 1. Постановка задачи

Medusa разрешает плагину/приложению указать React-компонент и **предопределённую самой Medusa** зону врезки (`product.details.after`, `order.list.before`, …). Своих зон завести нельзя.

Нам же нужно обратное: мы строим собственные admin-страницы (provider-settings, будущий integration-core UI), объявляем в них **свои** зоны (`provider-settings.form.after`, `integration.detail.side.before`, …), и хотим, чтобы любой сторонний пакет мог положить туда свой виджет — не трогая наш код.

В прошлом проекте это решалось `@gorgo/medusa-vite-plugin` (см. §3). Вопрос документа: **повторять тот подход или сделать иначе.**

---

## 2. Что предлагает Medusa нативно (факты)

| Механизм | Что даёт | Ограничение для нас |
|---|---|---|
| `defineWidgetConfig({ zone })` + default export, файл в `src/admin/widgets/**` | Виджет врезается в зону Medusa | Зоны — **фиксированный enum**; в наши страницы попасть нельзя |
| `defineRouteConfig({ label })`, файл в `src/admin/routes/**` | Новая **страница** в навигации админки | Гранулярность «страница», а не «точка внутри нашей формы» |
| Сборка админки | `@medusajs/admin-vite-plugin` собирает `src/admin/**` из приложения и всех плагинов в **один** SPA-бандл | Всё, что **вне** `widgets/`/`routes/`, в бандл сам не попадает |

**Вывод:** нативного API «зарегистрировать свою зону для третьих лиц» у Medusa нет. Любая кросс-пакетная врезка в наши страницы — это либо (а) перенаправление через нативный механизм виджетов/роутов, либо (б) собственная build-time-генерация статических импортов.

---

## 3. Как было сделано раньше (разбор `@gorgo/medusa-vite-plugin`)

Из скомпилированного `packages/utils/medusa-vite-plugin/dist/index.js`:

1. **Источники.** Плагину передаётся `options.sources` — список npm-пакетов. Для каждого краулится директория `./node_modules/<source>/.medusa/server/src/gorgo-widgets`.
2. **Парсинг.** Каждый `.tsx/.jsx/.ts/.js` (кроме `.d.ts` и `_`-папок) парсится babel'ом. Файл считается виджетом, если у него есть **default export** и `config` с полем `zone` (named export, либо bundled-вариант через `VariableDeclarator`/`exports.config`).
3. **Валидация зоны.** `zone` сверяется с **хардкод-списком собственных зон** (`MARKETPLACE_INJECTION_ZONES`, `MARKETPLACE_EVENT_INJECTION_ZONES`). Невалидные — отбрасываются с логом.
4. **Кодогенерация.** Собирается виртуальный модуль `virtual:gorgo/widgets`, экспортирующий `{ widgets: [{ Component: WidgetComponentN, zone: [...] }] }` со статическими `import`-ами по нормализованным путям. Есть хэш-инвалидция (md5 по содержимому `config` + default export) для dev-режима.
5. **Рендер.** Host-расширение админки импортирует виртуальный модуль и рендерит компоненты в своих кастомных местах (наш аналог `<InjectionZone>`).

Конвенции из `medusa-plugins/CLAUDE.md`: `src/gorgo-widgets/` — файлы виджетов для **гороговских** зон; `src/admin/widgets/` — обычные виджеты Medusa; `src/admin/components/gorgo-widgets/` — их компоненты.

По сути это **копия пайплайна `@medusajs/admin-vite-plugin`, но под наши зоны**. Работает; цена — отдельный build-инструмент и привязка к layout'у `.medusa/server`.

---

## 4. Почему descriptor.`customSections` из 2026-05-28 не закрывает задачу

`customSections: [{ component: () => import("./admin/x") }]` — литеральный спецификатор импорта **внутри собственного admin-бандла плагина-владельца descriptor'а**. Это работает для «плагин X дополняет свою же страницу X». Но:

- descriptor резолвится **на сервере** (наш loader), а компонент нужен **на клиенте** (в admin-бандле) — серверная декларация не может «принести» чужой React-компонент в клиентский граф;
- врезка в **чужую** зону (виджет от пакета B в форму пакета A или в общую страницу) требует, чтобы импорт компонента B оказался в admin-графе — а сам по себе он туда не попадёт.

Значит, нужен либо build-time-генератор статических импортов (§5.A), либо runtime-регистрация через уже-импортируемый Medusa-механизм (§5.C). Чистым расширением descriptor'а задача не решается.

---

## 5. Варианты

### Вариант A — Свой Vite-плагин + виртуальный модуль (повтор прошлого подхода)

Обобщить `@gorgo/medusa-vite-plugin`: вынести список зон из хардкода в опции (`zones: GorgoZone[]`), оставить всё остальное. Потребитель добавляет плагин в Vite-конфиг админки (через `medusa-config.ts → admin.vite`, как в прошлом проекте).

```ts
// medusa-config.ts (приложение потребителя)
admin: {
  vite: (config) => {
    config.plugins.push(gorgoVitePlugin({
      sources: ["@gorgo/medusa-payment-robokassa", "@acme/medusa-some-integration"],
      zones: GORGO_INTEGRATION_ZONES,   // экспортируем из @gorgo/medusa-integration
    }))
    return config
  },
}
```

```tsx
// файл стороннего пакета: src/gorgo-widgets/my-widget.tsx
export const config = { zone: "provider-settings.form.after" }
export default function MyWidget({ pluginId, instanceId }) { /* ... */ }
```

```tsx
// в нашей admin-странице
import gorgo from "virtual:gorgo/widgets"
<GorgoZone id="provider-settings.form.after" widgets={gorgo.widgets} ctx={{ pluginId, instanceId }} />
```

**Плюсы**
- Проверено в бою; DX совпадает с Medusa (`config.zone` + default export — разработчики уже умеют).
- Build-time → нулевая runtime-стоимость, статические импорты, tree-shaking, типобезопасность зон.
- Поддерживает раздельно опубликованные пакеты без общего рантайма.
- Можно почти буквально переиспользовать существующий код.

**Минусы**
- Потребитель обязан прописать плагин в Vite-конфиг админки — разовая настройка, но это трение и нагрузка на поддержку.
- Привязка к layout'у `.medusa/server/src/gorgo-widgets` — хрупко между версиями Medusa и при разных схемах hoisting'а (pnpm/yarn workspaces).
- Зависимости babel/fdir/magic-string и собственный build-инструмент в обслуживании.
- `sources` нужно перечислять вручную (нет авто-discovery установленных интеграций).

### Вариант B — Расширение через UI Routes (чисто нативно, без своего тулинга)

Не врезаться внутрь страниц, а дать каждой интеграции **свою страницу/таб** под нашим разделом настроек средствами нативных `defineRouteConfig`. Наша навигация просто перечисляет зарегистрированные провайдеры (данные уже есть из descriptor-API), а «углублённый» UI каждая интеграция реализует как обычный Medusa-роут.

**Плюсы**
- Ноль кастомного тулинга, только публичные API Medusa.
- Невозможно «сломать» сборку — Medusa сама всё собирает.
- Максимальная устойчивость к смене версий Medusa.

**Минусы**
- Гранулярность «страница», а не «точка внутри формы» — заявленный сценарий (виджет в нашей generated-форме, в `*.side.*`, рядом с секцией) **не закрывается**.
- Фрагментирует UX: вместо единой страницы провайдера — россыпь отдельных страниц.
- Не даёт нескольких независимых виджетов в одной зоне от разных пакетов.

Годится как дополнение (крупные кастомные экраны), но не как основной механизм врезки.

### Вариант C — Runtime-реестр через side-effect-регистрацию, бутстрапнутую нативным виджетом

Без своего build-инструмента. `@gorgo/medusa-integration` экспортирует singleton-реестр:

```ts
// @gorgo/medusa-integration/admin
const registry = new Map<GorgoZone, GorgoWidget[]>()
export function registerGorgoWidget(zone: GorgoZone, w: GorgoWidget) { /* push */ }
export function useGorgoWidgets(zone: GorgoZone) { /* read */ }
```

Каждая интеграция кладёт **один обычный Medusa-виджет** в `src/admin/widgets/`, который при импорте модуля регистрирует свои компоненты:

```tsx
// src/admin/widgets/gorgo-register.tsx
import { defineWidgetConfig } from "@medusajs/admin-sdk"
import { registerGorgoWidget } from "@gorgo/medusa-integration/admin"

registerGorgoWidget("provider-settings.form.after", { Component: MyWidget })

export const config = defineWidgetConfig({ zone: /* любая существующая зона-носитель */ })
export default () => null   // сам по себе ничего не рисует
```

Поскольку Medusa собирает все `src/admin/widgets/**` в один бандл и реестр — общий singleton, наши страницы читают его и рендерят.

**Плюсы**
- Ноль кастомного Vite-плагина, babel, краулинга, правки admin-конфига.
- Только публичные API Medusa; устойчиво к смене layout'а сборки.
- Авто-discovery: достаточно, чтобы интеграция была установлена (её виджет Medusa подхватит сама).

**Минусы / риски**
- **Тайминг регистрации** — ключевой риск. Зависит от того, **импортирует ли Medusa модули виджетов сразу при загрузке бандла** (тогда side-effect-регистрация надёжна) или **лениво** при заходе на страницу-носитель (тогда наша страница может отрендериться раньше регистрации). **Требует проверки** на текущей версии Medusa.
- Семантически «грязно»: нативный виджет используется как загрузчик, привязанный к произвольной зоне-носителю.
- Нет статической типизации соответствия «зона ↔ контекст», сложнее tree-shaking.

## 6. Сравнение

| Критерий | A. Vite-плагин | B. UI Routes | C. Runtime-реестр |
|---|---|---|---|
| Врезка внутрь наших форм/секций/`*.side.*` | ✅ да | ❌ нет (только страницы) | ✅ да |
| Несколько виджетов в одной зоне от разных пакетов | ✅ | ❌ | ✅ |
| Кастомный build-инструмент | да (babel/fdir) | нет | нет |
| Правка admin-конфига потребителем | да | нет | нет |
| Авто-discovery установленных интеграций | нет (`sources` вручную) | частично | ✅ |
| Привязка к внутренностям Medusa | layout `.medusa/server` | минимум | тайминг импорта виджетов |
| Runtime-стоимость / tree-shaking | лучшее | хорошее | хуже |
| Зрелость / проверено | ✅ в прошлом проекте | ✅ нативно | ⚠️ требует проверки тайминга |

## 7. Рекомендация (предварительная)

**Основной — A (Vite-плагин), но в облегчённой и обобщённой форме.** Это единственный вариант, который (а) даёт настоящую врезку в наши зоны, (б) уже проверен, (в) совпадает по DX с Medusa. Облегчить относительно прошлой версии:
- вынести зоны в опции/реестр из `@gorgo/medusa-integration` (убрать marketplace-хардкод);
- задокументировать одну строку подключения в `admin.vite`;
- по возможности заменить ручной `sources` на чтение из descriptor-loader'а (список установленных интеграций мы и так знаем на сервере — можно отдавать его плагину).

**Дополнительно — B** для крупных самостоятельных экранов интеграции (нативные UI Routes под нашим разделом). A и B не конфликтуют.

**C держать как запасной**: если проверка покажет, что Medusa импортирует модули виджетов **сразу** (а не лениво), C становится самым дешёвым по интеграции и снимает требование править admin-конфиг — тогда стоит пересмотреть выбор в его пользу. **Действие: подтвердить порядок импорта виджетов в текущей Medusa.**

## 8. Как это ляжет в 2026-05-28 (черновик правок, пока не применять)

1. **§3 descriptor** — добавить поле `widgets?` рядом с `customSections`:
   ```ts
   widgets?: Array<{ zone: GorgoZone; component: () => Promise<{ default: React.ComponentType<GorgoWidgetProps> }> }>
   ```
   и оговорить: `customSections` = врезка в **свою** форму; `widgets` = врезка в **любую** нашу зону (механизм доставки — §5, выбранный вариант).
2. **§8 Admin UI** — добавить подраздел «Кастомные зоны и сторонние виджеты»: список зон (`provider-settings.list.*`, `provider-settings.form.*`, `provider-settings.section.*`, задел `integration.detail.*`), компонент `<GorgoZone>` и контракт `GorgoWidgetProps` (`{ pluginId, instanceId, kind }`).
3. **§10 структура пакета** — добавить экспорт `@gorgo/medusa-integration/admin` (реестр + `<GorgoZone>` + типы зон); при выборе A — отдельный пакет `@gorgo/medusa-integration-vite-plugin`.
4. **§12 открытые вопросы** — добавить: «порядок импорта виджетов Medusa (блокер выбора между A и C)» и «как доставлять список установленных интеграций в Vite-плагин (ручной `sources` vs из descriptor-loader)».

## 9. Открытые вопросы

1. **Тайминг импорта виджетов Medusa** — eager или lazy? Прямо определяет жизнеспособность варианта C.
2. **Поддерживает ли текущая Medusa `admin.vite`-хук** для добавления нашего Vite-плагина так же, как в прошлом проекте? Подтвердить актуальный механизм.
3. **Контекст зоны.** Какой проп получает виджет (`pluginId`, `instanceId`, `kind`, текущее значение формы)? От этого зависит полезность врезки в `form.*`-зоны.
4. **Набор зон v1.** Минимально достаточный список (скорее всего только `provider-settings.form.after` + `provider-settings.list.after`), остальное по мере надобности.
5. **Изоляция ошибок.** Падение стороннего виджета не должно ронять нашу страницу — нужен error-boundary вокруг `<GorgoZone>`.
