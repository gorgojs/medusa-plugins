# Методы обмена данными: ERP/CRM интеграции

| Интеграция | Инициатор обмена | Методы обмена и реализация в Medusa |
|---|---|---|
| **1C:Enterprise** | **Только 1C** — сайт полностью пассивен. 1C по расписанию (каждые N минут) инициирует всю сессию от авторизации до завершения. Medusa не может вызвать 1C. | **Протокол:** CommerceML 2 XML через HTTP (не REST). Stateful-сессия из 4 шагов: `checkauth` → `init` → `file` (загрузка XML) → `import`. **Обмен заказами** (2-й блок): 1C вызывает `GET /query` — сайт возвращает новые заказы в XML; 1C вызывает `POST /file` — отправляет обновлённые статусы обратно. **Реализация в Medusa:** кастомный HTTP endpoint (`/hooks/1c/:id/exchange`) в `api/hooks/`; Medusa Workflow для обработки `import.xml` (категории + товары) и `offers.xml` (варианты + цены + остатки); кастомный XML endpoint для выдачи заказов на `GET /query`. **Аутентификация:** HTTP Basic на `checkauth`, session cookie на остальных шагах. |
| **МойСклад** | **Двусторонний.** ① *Medusa → МойСклад* — инициирует Medusa: по расписанию или по событию вызывает REST API МойСклад. ② *МойСклад → Medusa* — инициирует МойСклад: отправляет webhook-POST на зарегистрированный URL при изменении сущностей (CREATE / UPDATE / DELETE). | **Протокол:** JSON REST API 1.2 (`api.moysklad.ru/api/remap/1.2`). Полный CRUD через GET / POST / PUT / DELETE. **Сущности:** товары (`entity/product`, `entity/variant`), заказы (`entity/customerorder`), клиенты (`entity/counterparty`), остатки (`report/stock/all/current`), цены (`entity/pricetype`), склады (`entity/store`). **Webhooks:** регистрируются через REST (`POST /entity/webhook`); типы событий: `CREATE`, `UPDATE`, `DELETE` для всех ключевых сущностей. Заголовок `X-Lognex-WebHook-Disable` подавляет webhook для конкретного вызова. **Реализация в Medusa:** Medusa Job (scheduled) для периодического pull; Medusa Subscriber (`order.placed` и т.д.) для push в МойСклад; HTTP endpoint (`/hooks/moysklad/:id`) для приёма входящих webhooks; Medusa Workflow для оркестрации синхронизации. **Аутентификация:** HTTP Basic или Bearer token. |
| **RetailCRM** | **Двусторонний с ограничениями.** ① *Medusa → RetailCRM* — инициирует Medusa: Subscriber на события Medusa вызывает REST API RetailCRM. ② *RetailCRM → Medusa* — инициирует RetailCRM через **integration module callbacks** (не generic webhooks): вызывает зарегистрированные URL в конкретных точках (`onReserve`, `onOrderProductList`, `onProduct`). Пользовательские Triggers в UI ретайлCRM также могут генерировать исходящие события. | **Протокол:** JSON REST API v5 (`{subdomain}.retailcrm.ru/api/v5`). POST с `application/x-www-form-urlencoded` + JSON-строки для вложенных структур. **Сущности:** заказы (`/orders` — полный CRUD, история, комбинирование), клиенты (`/customers`), остатки (`/store/inventories` — чтение + загрузка по складу), товары (read-only ссылки из подключённых магазинов). **Callbacks:** регистрируются через `POST /api/v5/integration-modules/{code}/edit`; RetailCRM вызывает объявленные URL при резервировании, открытии карточки товара/заказа. **Реализация в Medusa:** Medusa Subscriber для `order.placed` → POST `/api/v5/orders/create`; HTTP endpoint (`/hooks/retailcrm/:id`) для приёма callbacks; Medusa Workflow для маппинга статусов. Ограничение: нет generic webhooks на произвольное изменение заказа — только через Triggers или polling. **Аутентификация:** API key как GET/POST параметр `apiKey`. Лимит: 10 req/s. |
| **Bitrix24** | **Двусторонний.** ① *Medusa → Bitrix24* — инициирует Medusa: вызывает REST методы Bitrix24 через incoming webhook URL или OAuth token. ② *Bitrix24 → Medusa* — инициирует Bitrix24: отправляет POST на зарегистрированный URL при наступлении CRM-событий (deals, contacts, orders, catalog). | **Протокол:** JSON REST API (`{account}.bitrix24.com/rest/{method}`). Поддерживает GET и POST. **Два контура:** CRM (`crm.*` — сделки, контакты, компании, лиды) и интернет-магазин (`catalog.*` + `sale.*` — товары, SKU, склады, остатки, заказы, оплата). **Outgoing webhooks:** события CRM — `ONCRMDEALUPDATE/ADD/DELETE`, `ONCRMCONTACT*`, `ONCRMCOMPANY*`; события каталога и заказов (`sale.*`). Регистрируются в Developer Resources Bitrix24. **Реализация в Medusa:** Medusa Subscriber для `order.placed` → вызов `crm.deal.add` или `sale.order.add`; HTTP endpoint (`/hooks/bitrix24/:id`) для приёма outgoing webhooks; Medusa Job для периодической синхронизации каталога/остатков; Medusa Workflow для маппинга сделок/заказов. **Аутентификация:** ① Incoming webhook — секретный ключ в URL (постоянный); ② OAuth 2.0 — `access_token` (1 ч) + `refresh_token` для приложений. Облако vs On-premise: API одинаковое, набор доступных полей каталога может отличаться. |
| **amoCRM / Kommo** | **Двусторонний.** ① *Medusa → amoCRM* — инициирует Medusa: Subscriber на события Medusa вызывает REST API amoCRM. ② *amoCRM → Medusa* — инициирует amoCRM: отправляет POST на зарегистрированный URL при изменении сделок, контактов, клиентов, задач, элементов каталога. | **Протокол:** JSON REST API v4 (`{account}.kommo.com/api/v4`). **Сущности:** сделки/лиды (`/leads`), контакты (`/contacts`), компании (`/companies`), клиенты B2C (`/customers`), каталоги-списки (`/catalogs/{id}/elements` — продуктовые списки без остатков и складов). **Важное ограничение:** нет нативной сущности «заказ» и нет управления остатками/ценами — amoCRM является CRM для воронки продаж, не e-commerce платформой. Интеграция целесообразна для post-sale CRM (лиды, клиенты, коммуникации), но не для синхронизации каталога или складского учёта. **Webhooks:** amoCRM → ваш URL при событиях: `add/update/delete` для leads, contacts, companies, customers, tasks, catalog elements. Ответ должен вернуться за 2 сек; retry: 5 мин → 15 мин → 15 мин → 1 час. Требует тариф Advanced/Pro/Enterprise. **Реализация в Medusa:** Medusa Subscriber для `order.placed` → создание сделки через `POST /api/v4/leads`; HTTP endpoint (`/hooks/amocrm/:id`) для приёма webhooks при изменении сделок; Medusa Workflow для маппинга статусов сделки → статус заказа Medusa. **Аутентификация:** OAuth 2.0 only (`access_token` краткосрочный + `refresh_token`). Лимит: 7 req/s. |
